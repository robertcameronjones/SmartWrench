Guidepoint Maintenance 2.0 Agentic

Rollout Plan

From today's working prototype to a production system serving real dealers and customers.

We have a working end‑to‑end prototype: voice + SMS, a real Twilio number, the LLM behind Kate, the case lifecycle worked out and signed off in concept. This plan moves us from "runs on a laptop with one operator" to "runs in the cloud for a pilot dealer's customers." Eight phases, roughly sequential.

Phase 1 — Internal Demo

Phase 1
Internal demo
~1 day
Show leadership and sales the working prototype end‑to‑end so they can see and feel what we'll be selling.
What we show
  • Voice booking call (real ElevenLabs voice)
  • SMS booking, including reschedule
  • The lifecycle one‑pager (this companion document)
Deliverables
  • 30‑minute demo deck (3 slides + screen share)
  • Recording for sales enablement
Owners
Product + engineering lead
Exit criteria
Leadership says "yes, build the rest."

Phase 2 — Lifecycle Signoff

Phase 2
Lifecycle signoff
~3 days
Walk Operations, Sales, and a friendly dealer through the lifecycle one‑pager. Lock the policy decisions before any production code is written.
Decisions to lock
  • Trigger sources (recall, DTC, regular maintenance)
  • Channel preference rule (per customer? per trigger type?)
  • Timeout windows (today: 24h nudge / 48h close)
  • Reschedule mechanics (slot inventory, blackout windows)
  • What "Decline" means commercially (do not contact list?)
Open questions
  • How do dealers actually feed us their open slots?
  • Who owns the customer "do not contact" master record?
  • Compliance: TCPA opt‑in language for SMS?
Owners
Product + Operations + a friendly dealer GM
Exit criteria
Lifecycle one‑pager v1 signed off; open items become Phase 3 prerequisites or explicit deferrals.

Phase 3 — Deploy Prototype

Phase 3
Deploy prototype to cloud
~3 days
Get the simulator off the laptop and onto a real cloud URL with secure access, so multiple people can use it from anywhere without our laptop being on.
Stack
  • Render (or Fly.io) for the simulator service
  • Cloudflare Access in front for login + 2FA (free up to 50 users)
  • Custom subdomain (e.g. kate‑demo.guidepoint.com)
  • Twilio webhook permanently pointed at the new URL
What changes in code
  • Persistent storage (small disk or SQLite file)
  • Twilio signature validation back ON
  • Bearer‑token auth on internal endpoints
  • Log retention policy
Owners
Engineering
Exit criteria
Anyone on the access list can log in from anywhere, fire a case, complete an SMS booking. Twilio resilient to laptop closing.

Phase 4 — Find Pilot Dealer

Phase 4
Find pilot dealer
~2–4 weeks (sales cycle)
Identify and onboard one friendly dealer willing to feed us live triggers and live customer phone numbers.
What we need from them
  • Customer roster (name, phone, vehicle, opt‑in status)
  • Open service slot inventory (manual feed OK initially)
  • One SMS opt‑in plan that covers their customers (TCPA compliance)
  • A named operator to watch the queue and approve first few cases
What they get
  • Custom branded outreach (their dealership name, voice persona)
  • Read‑only dashboard for cases in flight
  • Pilot pricing (free or steeply discounted)
Owners
Sales + Product
Exit criteria
Signed pilot agreement. First 10 real cases run through the system.

Phase 5 — Concurrency: Multiple Parallel Cases

Phase 5
Concurrency & durability
~1 week
Handle dozens of cases at the same time without losing state, double‑sending, or interleaving turns within a single conversation.
Engineering work
  • Move case + conversation + history + routing into a single SQLite (then later Postgres) schema
  • Per‑conversation row lock so simultaneous inbounds serialize
  • Background scheduler for time‑driven phase transitions (reminder, verify nudge, timeouts)
  • Trigger queue with the 1:1‑per‑phone rule enforced
  • Idempotency keys on outbound SMS so a retry can't double‑send
Test plan
  • Load test: 100 simultaneous open cases
  • Crash test: kill the service mid‑conversation, restart, customer's next reply is handled cleanly
  • Drift test: clock advance through full lifecycle including reminders & verify
Owners
Engineering
Exit criteria
Pilot dealer's full daily volume runs through the system with zero lost or duplicated messages over a week.

Phase 6 — Bare‑Bones Reporting

Phase 6
Reporting v1
~1 week
Give Operations and the dealer the numbers they need to trust the system.
Operator dashboard
  • Open cases (by phase)
  • Queued triggers
  • Today's outcomes (booked / declined / timed out / no‑show)
  • Channel mix (voice vs SMS)
  • Inspect any case → see full transcript
Dealer report (weekly)
  • Triggers received → cases opened → booked → showed → serviced
  • Conversion at each phase
  • Average time‑to‑book
  • $ value of booked work (if dealer feeds us pricing)
Owners
Engineering + Operations
Exit criteria
Dealer GM uses the dashboard unprompted at least once a week.

Phase 7 — Go‑Live with Pilot

Phase 7
Pilot go‑live
~4 weeks (live operation)
Run real customers, learn, iterate prompts and policies, build the case study.
What "running" looks like
  • Daily trigger feed from the dealer (CSV upload OK at first)
  • Operator monitors the queue, intervenes on edge cases
  • Weekly review with the dealer GM on metrics
  • Bi‑weekly prompt tuning based on transcripts
Owners
Product + Operations + dealer operator
Exit criteria
30 days of clean operation, conversion numbers we can quote, two case studies in writing.

Phase 8 — Implementation Handoff

Phase 8
Implementation handoff to engineering org
~2 weeks
Transition from prototype team to the engineering organization that will own this in production at scale.
Deliverables
  • Architecture document (this lifecycle + data model + service map)
  • Runbook (how to start/stop, where logs live, what alerts mean)
  • Test suite (unit + integration + smoke)
  • Migration path from SQLite/Render to AWS production stack (next section)
  • Backlog of known limitations + next features
Owners
Engineering lead → receiving team's tech lead
Exit criteria
Receiving team can deploy a fix, run the test suite, and respond to a Twilio outage without prototype team's help.

What it takes to stand up on AWS for production

Render or Fly.io is right for the prototype and pilot. When Guidepoint is ready to host this at production scale on the existing AWS footprint, here's what's added or replaced.

Compute

Run the FastAPI app behind a load balancer with multiple instances for HA.

  • ECS Fargate (or EKS if there's an existing Kubernetes presence)
  • ALB (application load balancer) with HTTPS
  • Auto‑scaling on request volume

Data

Replace SQLite with a managed Postgres for concurrency, durability, and backups.

  • RDS Postgres (Multi‑AZ for HA)
  • Daily snapshots + point‑in‑time recovery
  • S3 for transcript exports / long‑term audit storage

Networking & DNS

  • Route 53 for the production hostname
  • ACM certificate for HTTPS
  • VPC with private subnets (DB), public subnets (ALB only)

Secrets & identity

  • Secrets Manager: Twilio token, OpenRouter key, ElevenLabs key, DB password
  • IAM roles per service (no shared keys in env)
  • SSO or Cloudflare Access for operator dashboard access

Observability

  • CloudWatch logs (structured JSON)
  • CloudWatch alarms: webhook 5xx rate, LLM latency p95, queue depth
  • Optional: Datadog / Honeycomb for richer tracing

Compliance & safety

  • SMS opt‑in records stored alongside customer (TCPA defensibility)
  • PII redaction in logs and transcripts (phone, name)
  • Data retention policy + automated purge
  • Per‑dealer data isolation (row‑level security in Postgres)

CI/CD

  • GitHub Actions → build container → push to ECR → deploy to Fargate
  • Staging environment that mirrors production
  • Database migration tooling (Alembic)

Operational extras

  • Twilio messaging service (vs single number) for high volume + opt‑out compliance
  • ElevenLabs production tier (vs trial)
  • OpenRouter or direct provider account with billing alerts
  • On‑call rotation + PagerDuty integration
