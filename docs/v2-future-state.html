Internal · Engineering plan brief

Guidepoint Maintenance — v2 Future State

The case lifecycle, the architecture, and the code layout we're building toward.

v2 takes the unified case-lifecycle foundation from v1 (SMS through the same CaseManager as voice) and grows it into the full maintenance flow: trigger → outreach → dealer confirmation → booking → reminder → day-of → service → feedback → close. It does this with two distinct state machines, each with its own owner, behind one published Protocol surface between them.

Two state machines, one Protocol boundary

The case has its own state machine, and each call (voice or SMS) has its own state machine. Neither knows the other's internals. They meet at the CallSession Protocol; their only contract is "ask for a call, get back a CallOutcome."

The split is temporal. Outside a call, the CaseManager is in charge and the case sits in a durable stage (e.g. INITIAL_REMINDER_DUE). When it's time to talk to the customer, the CaseManager hands control to a CallManager, which advances its own per-call state turn by turn (dial → opened → conversing → terminal). When the call ends, the CallManager returns a single CallOutcome and control flows back to the CaseManager, which records the outcome and transitions the durable case state (e.g. to BOOKED). The CaseManager is always the system of record; the CallManager is the active driver during its window.

CaseManager

System of record. Owns durable case state across calls.

  • Tracks lifecycle stage: CREATED → ... → COMPLETED
  • Reacts to business hours, geofence, reminder windows, dealer responses, opt-out
  • Hands off to a CallManager when the case needs to talk to the customer
  • On call return: records the CallOutcome and advances the case stage
  • Never sees: LLM turns, twilio SIDs, ElevenLabs status, stop keywords

CallManager (per channel)

Active driver during a call. Owns the per-call state machine.

  • Has exclusive ownership while the call is live — case state is frozen
  • Advances per-call state: dial → opened → conversing → terminal
  • Reacts to LLM turns, twilio inbound, stop keywords, inactivity
  • Returns one CallOutcome when the conversation ends
  • Never sees: reminder timers, dealer state, geofence, lifecycle stage names

The boundary is enforced by the CallSession Protocol that already exists today. The CaseManager imports the Protocol, never the implementation.

class CallSession(Protocol):
    async def start(self, *, case: Case, intent: Intent) -> CallOutcome: ...

# Voice and SMS each have their own CallManager that satisfies this Protocol.
# Inside each, the channel runs its own pure reducer + thin driver — but
# none of that complexity leaks past the Protocol.

Prompt composition & channel strategy

Invariant: one call = one stage = one prompt. This is what makes ElevenLabs' single-prompt constraint a non-issue and keeps the two channels symmetric end-to-end.

During a call, the active CallManager composes its prompt once at call-start and uses it for the entire conversation. Stage transitions (e.g. INITIAL_REMINDER_SENT → FINAL_REMINDER_DUE, SHOWED → AWAITING_FEEDBACK) happen between calls, never within them. We never swap a prompt mid-conversation.

Channel × stage matrix

StageVoiceSMS
OUTREACHyesyes (customer's choice)
INITIAL_REMINDER—yes
FINAL_REMINDER—yes
FEEDBACK—yes

Voice is used only for initial outreach; everything downstream is SMS. This is the single design choice that lets us live within ElevenLabs' constraint without effort: the voice agent only ever needs one prompt and one tool set, both for OUTREACH.

Why this works for both channels

SmsCallManager

Channel where we own the LLM call.

  • Calls build_prompt(case, stage) once at conversation start
  • Caches the result for the call's lifetime
  • Sends the same prompt with every LLM turn
  • Per-stage allowed tool list — trivial to vary, we control the call

VoiceCallManager

Channel where ElevenLabs owns the LLM call.

  • Calls build_prompt(case, OUTREACH) once at agent-start
  • Hands the string to ElevenLabs; ElevenLabs locks it for the call
  • Never swaps mid-call (matches ElevenLabs' constraint exactly)
  • Tools configured once at the agent level (book_slot, decline, reschedule_initial)

The composer doesn't know about channels

# prompt_composer signature — channel-agnostic
def build_prompt(case: Case, stage: Stage) -> str: ...

# Internally:
#   core (identity, tone, guardrails)
# + stage overlay (per-stage instructions + tool surface)
# + case variables (customer, vehicle, slot, etc.)
# ----------------------------------------
# = one composed string

Same function, same inputs, same output, regardless of who's about to use it. The channel concern lives entirely in the CallManagers: voice composes once and hands off; SMS composes once and caches. The composer doesn't care which.

What we lose by this discipline

  • We can't mid-conversation re-prompt based on what the customer said in turn 3. (We don't want to — it breaks conversational continuity and would surprise the LLM mid-flight.)
  • We can't run voice for INITIAL_REMINDER / FINAL_REMINDER / FEEDBACK without configuring additional ElevenLabs agents. (We're not doing this — voice is OUTREACH only by design.)

If either of those constraints ever needs to flex, the CallSession Protocol is still the only seam that has to change.

Queue inventory & backpressure

Every async boundary in the system has a bounded queue with explicit overflow behavior and a runtime-checkable depth accessor. No silent unbounded queues — overflow either logs and drops (telemetry-shaped) or logs and aborts (control-flow-shaped), but never just grows memory.

The discipline: control signals (CaseSignals, customer turns) get bounds that abort or yield when overflowed because dropping one silently breaks correctness. Telemetry (EventBus subscribers) drops on overflow because the publisher must never block on a slow consumer. Outbound API calls (Twilio, LLM) aren't queue-shaped — they're rate-limit-shaped, so they get concurrency semaphores and tenacity retries instead.

# Queue Where Max Overflow behavior Runtime check
1 EventBus subscriber queues guidepoint/events/_bus.py 256/sub put_nowait → log queue.overflow.event_bus, drop event. Publisher never blocks. live bus.subscriber_depths() → tuple[(int, int), ...]
2 SMS inbound per active case sms_adapter/_call_session.py 32 turns put_nowait → log queue.overflow.sms_inbound, return False to webhook. 24h inactivity timer is the safety net. live session.queue_depths() → dict[CaseId, (int, int)]
3 Per-case CaseSignal queue guidepoint/case/_driver.py (Phase 4) 64 signals put_nowait → log queue.overflow.case_signal, transition case to ABANDONED with detail="signal queue overflow". Signals are control flow — better to lose one case loudly than corrupt all of them silently. phase 4 driver.queue_depths() → dict[CaseId, (int, int)]
4 Twilio outbound concurrency sms_adapter/_twilio.py (Phase 5) 8 concurrent asyncio.Semaphore(8) — additional calls wait. Tenacity handles Twilio 429s with exponential backoff. phase 5 twilio.in_flight() → int
5 LLM completion concurrency sms_adapter/_litellm.py (Phase 5) 8 concurrent asyncio.Semaphore(8) — additional calls wait. Tenacity handles provider rate-limits. phase 5 llm.in_flight() → int

The "checkable" part

Every row's runtime-check column is an actual method on the live object. A small /health/queues endpoint on the simulator (Phase 10) calls each one and returns:

{
  "event_bus":   [(12, 256), (3, 256)],
  "sms_inbound": {"case_abc": (4, 32), "case_def": (1, 32)},
  "case_signal": {"case_abc": (2, 64), "case_def": (0, 64)},
  "twilio_in_flight": 2,
  "llm_in_flight": 1
}

This is the difference between "we documented the queues" and "we know the queues are healthy right now." Operators can poll the endpoint during user-group testing to spot a queue creeping toward its max before it actually overflows.

Overflow log discipline

Every overflow path emits a structured log line with the same shape:

queue.overflow.<queue_name>
  current_depth: int   # what the queue was at when overflow hit
  max_depth:     int   # the configured bound
  <queue-specific fields>  # e.g. case_id, payload_type, from_number
  level:         warning   (telemetry) | error (control-flow)

A single log scrape pattern (queue.overflow.) catches every backpressure event in the system. Phase 11 tests assert that each overflow path produces both the structured log AND the documented behavioral consequence (event dropped / call yields / case aborted).

Read this like an org chart

  • guidepoint.case is the brains — case state machine + Protocols. Everything else plugs into it.
  • sms_adapter and guidepoint.case.elevenlabs are the two channel implementations — they satisfy the CallSession Protocol but otherwise stay in their own lanes.
  • guidepoint.persistence holds backends; build_app picks one via env var. JSON stays usable for tests.
  • simulator is the operator UX — composes everything, gives operators a UI to fire triggers and emit world signals. Same signals as prod; the only difference is who's clicking.
  • prompt_composer grows a Stage enum so the right prompt + tool surface goes to the LLM for each case stage.

Discipline (the test cadence)

After every phase commit, four lines must be green before phase N+1 opens:

sms_adapter:     pytest -q  → X/X passing
simulator:       pytest -q  → Y/Y passing
guidepoint:      pytest -q  → Z/Z passing  (+ ≥90% coverage)
prompt_composer: pytest -q  → 15/15 passing

If anything regresses, it gets fixed in the same commit, not the next one. No phase opens until the prior phase is locked.
