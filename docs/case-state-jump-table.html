Retired / renamed values (legacy-migration keys only)

These are no longer declared on CaseState. They live only as keys in _LEGACY_STATE_MIGRATIONS so any fossil persisted row still validates instead of raising:

  • slot_proposed, slot_picked → migrate to contacting_customer. v1 outreach sub-states; v2 collapses all outreach into CONTACTING_CUSTOMER.
  • awaiting_feedback → migrate to showed. A feedback touchpoint that was specced but never wired; SHOWED goes straight to COMPLETED on geofence-out.
  • confirming_with_dealer → migrate to booked. Renamed, not deleted: the customer has picked a slot and it is confirmed, so the case is booked while it waits on the reminder timers. The old v1 terminal booked is gone — booked is now this live, non-terminal value.

Naming smell: *_SENT is the actionable send state, not a "done" state

The reducer transitions *_REMINDER_DUE → *_REMINDER_SENT before the message is composed or sent: the driver persists final_reminder_sent first, then dispatches the PlaceCall, and only then does the SMS composer read the case and hand the LLM case_state = final_reminder_sent. So the state the model sees while it is being asked to send the reminder is already named in the past tense ("sent").

Observed failure (kate-v2, case case_144ae743d013944b): the initial reminder composed correctly, but the day-of reminder — same mechanism, but now with a prior reminder already in the transcript and a ..._sent state name — led the model to the post-booking prompt's "any other state" escape hatch, which was then delivered to the customer verbatim ("Got it — the system isn't expecting input from me right now."). Two coupled defects: (a) a past-tense state used as the "do it now" trigger, and (b) a fallback branch that emits real customer-facing text with no "diagnostic, do not send" guard.

Cleaner shape: keep the case in *_REMINDER_DUE while composing/sending and only advance to *_REMINDER_SENT after the send succeeds (and key the post-booking prompt on *_due); or drop the DUE/SENT split for the actionable moment entirely.

Per-state transitions

Each block lists every non-ignored branch for that state. Branches not listed return ignored:<why> (no state change, no actions). The opt-out column applies only when CallEnded.outcome.business_outcome == "opted_out"; it routes via the same universal opt-out path as the cross-cutting CustomerOptedOut signal.

Voice vs SMS customer signals. Voice raises one CallEnded per dialed call carrying a business_outcome (booked, declined, confirmed, cancelled, rescheduled, opted_out, or inconclusive). SMS is turn-driven: each customer text arrives as InboundSmsReceived(from_phone, body, message_sid), and the reducer classifies the body into either (a) a vocabulary-matched business outcome — confirm / reschedule / cancel for reminders, digit pick / decline / cancel keyword for outreach — that routes through the same logic the voice path uses, or (b) free text, which keeps the case in the same state and emits a PlaceCall(stage=…) so the SMS dispatcher composes one LLM reply. SMS has no incomplete_retry path; the queue/worker is single-shot.

Why some SHOWED rows are marked "defensive"

The reducer routes SHOWED calls through the FEEDBACK call stage (_stage_for_state(SHOWED) == FEEDBACK) and accepts feedback / inconclusive CallEnded outcomes. But no v2 reducer branch ever emits PlaceCall(stage=FEEDBACK), so those branches only fire if something outside the v2 flow (legacy code, test setup, or operator action) places a feedback call. The happy path from SHOWED is VehicleExitedDealer going straight to COMPLETED.

Terminal states

StateReached via
DECLINEDVoice: CONTACTING_CUSTOMER + CallEnded(declined). SMS: any outreach-shaped state (CONTACTING_CUSTOMER, RESCHEDULING) + InboundSmsReceived with the N+1 "none of those work" digit or a CANCEL/CANCELLED/CANCELED keyword.
CANCELLEDVoice: INITIAL_REMINDER_SENT, FINAL_REMINDER_SENT, or RESCHEDULING + CallEnded(cancelled). SMS: *_REMINDER_SENT + InboundSmsReceived with cancel vocabulary (3/NO/N/CANCEL/…).
ABANDONEDInconclusive call at attempt_count == 2 in CONTACTING_CUSTOMER or RESCHEDULING. Voice-only — SMS has no inconclusive/retry path.
NO_SHOWFINAL_REMINDER_SENT + EndOfBusinessDayReached (no geofence-in by close of business).
RESCHEDULE_FAILEDCustomer requested a second reschedule at either reminder (reschedule_count == 1), via CallEnded(rescheduled) on voice or InboundSmsReceived with reschedule vocabulary (2/RESCHEDULE/…) on SMS.
OPTED_OUTCross-cutting CustomerOptedOut, or CallEnded(opted_out) from any non-terminal state.
COMPLETEDHappy path: SHOWED + VehicleExitedDealer. The audit trail gets both case.geofence.exited and case.closed.service_event_complete. Also reachable via the defensive feedback-call branches above.

The inbound SMS unhandled-state branch is intentionally not an ignored:* reason — it carries inbound_unhandled_at_{state} and emits a RecordEvent so operators can grep for surprise customer texts.
  • The reducer is pure: it returns actions; the driver applies them. Actions are: PlaceCall, RequestDealerSlots, RequestDealerConfirmation, ScheduleTimer, CancelTimer, RecordEvent.
  • Only PlaceCall ever causes an LLM call downstream, but how depends on the case's initial_channel:
    • Voice: CaseDriver._run_place_call → CallManager.place() → ElevenLabs agent (LLM runs inside the ElevenLabs session).
    • SMS: CaseDriver._run_sms_outbound → SmsDispatcher.dispatch_outbound() → SmsMessageComposer.compose() (LLM call) → outbound queue → Twilio. There is no per-case session loop; the next customer reply re-enters the reducer as a fresh InboundSmsReceived signal.
    Sliders (geofence, EoD, business hours) never produce PlaceCall. Inbound SMS produces a fresh PlaceCall only on the free-text and the booking-acknowledgement branches; the vocabulary-matched confirm / cancel / reschedule branches do not, since those transitions are silent (cancel/reschedule have downstream actions of their own, confirm at either reminder is a deliberate no-op).
  • OutboundDispatched is the queue worker's audit callback. It carries the real Twilio MessageSid (the assistant turn in HistoryStore was written with the queue's item_id as a placeholder handle). The reducer records it; it never drives a state transition.
