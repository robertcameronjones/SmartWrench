What's new in this revision (per your dictation)

  1. New state BOOKING_COMPLETE between CONFIRMING_WITH_DEALER and the reminders. DealerConfirmed lands here with no guard.
  2. *_DUE states re-oriented. INITIAL_REMINDER_DUE = "24-hour-prior timer has fired, reminder is owed." FINAL_REMINDER_DUE = "day-of timer has fired (dates now match), reminder is owed." Send queue owns delivery timing.
  3. *_SENT states handle only the decision-changing customer responses: reschedule, cancel, opt out. Confirm / silence is a no-op.
  4. No-show triggers a reschedule. When the end-of-day tick fires while still in FINAL_REMINDER_SENT with no geofence-in, the case transitions to RESCHEDULING. No NO_SHOW terminal state.
  5. Service event complete is the terminal happy path. SHOWED + geofence-out → SERVICE_EVENT_COMPLETE. AWAITING_FEEDBACK and COMPLETED are gone from the v2 state machine.
  6. No guards anywhere. The previous "date comparison" and "reschedule budget" guards are removed from the table. If we want to keep them as reducer-internal integrity checks, they live as defensive no-ops, not as transition branches.

PlaceCall held in the queue when CustomerOptedOut fires — The held call may still go out (small bounded risk you accepted). The opt-out closes the case in the reducer; the queue does not need to inspect case state to suppress in-flight items for v1.

Open questions you didn't address — flagging, not assuming

  1. Reschedule budget. The prior model capped reschedules at 1 (then routed to RESCHEDULE_FAILED). You didn't mention it. Drop the cap entirely (infinite reschedules), or keep it and pick a new terminal name?
  2. Early geofence-in. What if the vehicle pulls into the dealer while we're still in BOOKING_COMPLETE, INITIAL_REMINDER_DUE, or INITIAL_REMINDER_SENT — i.e., before the final reminder fires? Currently only FINAL_REMINDER_SENT + VehicleEnteredDealer → SHOWED is in the table.
  3. Same-day booking. If the customer books at 10 AM for 2 PM today, the 24-hour-prior timer's fire time is yesterday (already in the past). Skip arming it entirely, or arm it and let the timer service drop past fires? Either way, the day-of timer fires almost immediately.
  4. Opt-out from BOOKING_COMPLETE. Covered by the cross-cutting rule above (sends to OPTED_OUT). Confirming you want it to apply here too.
  5. Cancel during outreach / rescheduling. Today, an outreach CallEnded(outcome=cancelled) isn't listed because customers don't typically cancel before booking. Confirm we don't need a separate row.
  6. MessageDispatched event. This event doesn't exist in the code yet — it's a placeholder for whatever the send queue emits back to the reducer when an SMS actually goes out. Name + payload to be decided when we wire the send queue.
  • Timer arming happens as an action emitted when entering BOOKING_COMPLETE. Future-day booking arms both timers (24h-before + day-of). Same-day booking arms only the day-of timer.
  • Send queue owns business-hours gating. When a *_DUE state's PlaceCall action lands in the queue outside business hours, the queue holds it until open.
  • Reducer-internal integrity checks (e.g., "is this slot_id actually in case.offered_slots?") still exist as defensive no-ops but do not branch state transitions, so they don't appear in this table.
