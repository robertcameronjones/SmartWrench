# ADR 0003: TOML for human-authored agent config; `tomli_w` for serialization

- **Status:** Superseded by [ADR 0004](0004-json-everywhere.md) (2026-05-10)
- **Date:** 2026-05-10
- **Owner:** Rob Jones
- **Reviewers:** —

> **Superseded.** This decision lasted one cycle. See ADR 0004 for the
> reversal: parity with ElevenLabs' wire format outweighed TOML's
> comment-and-diff advantages for this project. Kept on file for the
> record.

## Context

The local `agent/` module is the source of truth for the ElevenLabs agent
configuration: name, LLM, voice, language, dynamic-variable defaults, and the
list of attached tools. A human reviews and edits these files; CI then `apply`s
them to the live agent. The format must be:

1. **Diffable** — small edits show small diffs.
2. **Comment-friendly** — humans annotate why a value is what it is.
3. **Stdlib-readable** — Python 3.11+ ships `tomllib` for parsing.
4. **Strictly schema'd** — every read goes through Pydantic with
   `extra="forbid"`.

JSON fails (1) and (2). YAML brings significant indentation footguns and a
sprawling spec (anchors, references, tag system) that we do not need.
Pydantic-in-code is right for the per-call `Case` schema (typed, validated,
versioned with code), but wrong for human-edited config (recompiling Python to
change a temperature is silly).

## Decision

- **TOML** for all human-authored agent configuration (`config/agent.toml`,
  `config/tools/<name>.toml`).
- **Markdown** for the system prompt body (`config/system-prompt.md`) — long
  prose belongs in markdown, not in a stringified TOML key.
- **JSON** for example call payloads / fixtures (`fixtures/cases/<id>.json`)
  because they are machine-generated by the trigger CLI and the live REST
  endpoint will produce JSON anyway.
- **Pydantic-in-code** for the `Case` schema and all other internal types.

For serialization we add one runtime dependency:

- `tomli_w` (a 200-line, single-purpose, zero-dependency library by the same
  authors as the stdlib `tomli`/`tomllib`). Reading uses stdlib `tomllib`.

`tomlkit` was considered for round-trip comment preservation but rejected:
larger surface area, slower, and we generate config from scratch on `pull` so
preservation is not required. Comments in human-edited files are preserved by
the obvious mechanism — humans don't run `pull` over their hand edits without
intent, and `pull` writes a banner header making generated files unmistakable.

## Consequences

- Config is round-trippable: `pull` writes TOML, the human edits, `apply`
  reads TOML and pushes to ElevenLabs.
- Dependency footprint grows by one library (`tomli_w`). It is pure-Python,
  has no transitive dependencies, and is widely deployed.
- Long-form prose (system prompt) lives in markdown so the LLM-facing text
  renders correctly in code review and editors.
- The `Case` schema remains in code; changing it requires a code change and
  passes through the same Pyright/Ruff/Pytest gates as everything else.

## Alternatives considered

- **YAML** — too much spec for too little benefit; security gotchas around
  `yaml.load`.
- **JSON for everything** — no comments, ugly for hand editing.
- **`tomlkit`** — over-spec'd for our needs; we don't need comment
  round-tripping.
- **Pydantic-in-code for agent config** — forces a code change for every
  prompt tweak; defeats the "edit, diff, PR" loop.
